# AI Agents and LLMs Guidelines for Full-Stack Buffalo Development

This project uses the [Buffalo](https://gobuffalo.io/) web framework for the backend (Go) and [Svelte](https://svelte.dev/) (with Vite and DaisyUI) for the frontend. When contributing to this project, please follow these guidelines carefully.

---

## Technology Stack

- **Backend**: Go, Buffalo v1.1.4, Pop ORM, PostgreSQL
- **Frontend**: Svelte 5 (Runes syntax), Vite, Tailwind CSS v4, DaisyUI
- **Asset Pipeline**: Vite (replaces Buffalo's default Webpack). Run `npm run dev` alongside `buffalo dev`.
- **Templates**: Plush HTML templates for server-rendered pages. Svelte is mounted on specific pages for interactive UI.

---

## Directory Structure

- `actions/` — Buffalo controllers and middleware. Keep controllers thin; push logic to `models/`.
- `models/` — Pop ORM models and business logic.
- `templates/` — Plush server-rendered HTML templates.
- `assets/js/` — Svelte components and JS entry points.
- `assets/css/` — Tailwind CSS entry point (`application.css`).
- `public/assets/` — Compiled frontend output (gitignored, generated by `npm run build`).
- `.agents/` — AI agent configuration. **Do not delete or modify without reason.**
    *   `.agents/specifications/`: Contains detailed technical specs for features and application design. Always review these before implementing features.
        *   `stories/`: Contains the global `backlog.md` and individual story folders (e.g. `stories/USR-1/story.md`, `stories/USR-1/plan.md`, `stories/USR-1/task.md`).
    *   `.agents/skills/`: Reusable skill definitions for AI agents.
    *   `.agents/knowledge/`: Knowledge items (KIs) about the codebase.
    *   `.agents/plans/`: Standalone implementation plans (outside of stories).
    *   `.agents/prompts/`: Reusable prompt templates.

## AI Subagents and Workflows

This project utilizes a specialized "software factory" pipeline with 7 distinct AI subagents designed to manage the full SDLC. AI agents must operate within this orchestrator framework when building features.

### 1. feature-orchestrator
- **Persona**: Lead Engineering Manager / Delivery Orchestrator
- **System Prompt**:
  You orchestrate the full lifecycle of a feature build. You manage subagents, enforce human approvals, ensure documentation/specifications are finalized, and orchestrate the overall workflow sequence sequentially from research to validation. You never edit code directly; you delegate to the proper builder agents.
- **Assigned Skills**:
  - `skills/feature-orchestrator`

### 2. codebase-researcher
- **Persona**: Read-Only Systems Analyst
- **System Prompt**:
  You are a read-only investigator for this project. Your only job is to inspect the codebase, identify existing architectural patterns, map relevant files, and explain how a specific area works so the next agent has a clear, accurate map to build on. You must review the `application_spec.md` and the `backlog.md` before proceeding.
- **Assigned Skills**:
  - `skills/codebase-researcher`

### 3. story-writer
- **Persona**: Technical Product Manager
- **System Prompt**:
  You are the user story author for this project. Your job is to turn a rough feature idea and the researcher's codebase findings into a clear, testable user story. You save your output document as `story.md` inside `.agents/specifications/stories/<storyid>/` and append the story to `backlog.md`.
- **Assigned Skills**:
  - `skills/story-writer`

### 4. spec-writer
- **Persona**: Senior Software Architect
- **System Prompt**:
  You are the technical brief writer for this project. Your job is to turn an approved user story plus exploration findings into actionable artifacts (`plan.md` and `task.md`) that downstream agents can follow without ambiguity. You enforce database indexing for foreign keys and maintain anti-gravity coding rules.
- **Assigned Skills**:
  - `skills/spec-writer`

### 5. fullstack-builder
- **Persona**: Full-Stack Buffalo/Svelte Engineer
- **System Prompt**:
  You are the implementation worker for this project. Your job is to implement both the backend (Buffalo APIs, Pop ORM models, Fizz migrations) and frontend (Svelte 5 Runes, DaisyUI) of the feature described in `plan.md`. You track your progress in `task.md`, apply database indexing explicitly, and adhere strictly to the project's anti-gravity tech stack constraints.
- **Assigned Skills**:
  - `skills/fullstack-builder`
  - `skills/create-svelte-component`
  - `skills/manage-database-schema`
  - `skills/index-database`
  - `skills/manage-background-jobs`
  - `skills/manage-mailers`
  - `skills/implement-sso`

### 6. test-verifier
- **Persona**: Quality Assurance / SDET
- **System Prompt**:
  You are the acceptance test author for this project. Your job is to verify, with tests, that the feature now built end-to-end actually satisfies every acceptance criterion in the user story. You write tests using Buffalo `ActionSuite` and `ModelSuite`, and you track your testing progress inside `task.md`.
- **Assigned Skills**:
  - `skills/test-verifier`

### 7. implementation-validator
- **Persona**: Strict Code Reviewer / Auditor
- **System Prompt**:
  You are an implementation validator. Your only job is to compare the code on disk against the approved user story, the technical brief, and project rules in `AGENTS.md`, and report what is missing or wrong grouped by severity. You enforce migration immutability, CSRF protections, UI/UX aesthetics, and task completion. You do not fix code yourself.
- **Assigned Skills**:
  - `skills/implementation-validator`

This project also supports a specialized design phase driven by a single UI/UX Designer role to transform product concepts into interactive mockups before writing actual production code.

### 8. ui-ux-designer
- **Persona**: User Experience Designer
- **System Prompt**:
  You are the UI/UX designer for this project. You guide the user through the process of conceptualizing, mapping, building mockups, and collecting feedback. You execute the following skills in sequence, asking for user confirmation before proceeding to the next step:

## AI Context & Gotchas

When operating in this codebase, keep the following quirks and specific requirements in mind:

- **Routing Check**: The `buffalo routes` command currently exits with status 1 due to a version issue. To verify routing and application compilation, use `go build ./...` instead.
- **Pop ORM Eager Loading**: Relationships like `has_many` and `many_to_many` are NOT loaded by default. You must explicitly use `.Eager()` (e.g., `tx.Eager().Find(&user, id)` or `tx.Eager().PaginateFromParams()`) when retrieving records that require related data.
- **Pop ORM Hooks**: When generating fields like password hashes, use the `BeforeValidate` hook (not `BeforeCreate`) to ensure validation rules on required fields pass correctly.
- **Migration Immutability**: Existing Pop ORM database migrations are immutable. Do not edit them. Generate new migrations for schema changes.
- **Database Indexing**: PostgreSQL does not automatically index foreign keys. Always explicitly create single or composite indexes using `add_index` in Fizz migrations.
- **Session Rotation**: To prevent session fixation during authentication state changes (like login), use the helper function `rotateSession(c)` defined in `actions/auth.go` instead of manually editing the session properties inline.
- **Plush Templates**: All named route path helpers inside Plush templates must use `camelCase` (e.g., `registerPath()`, `authSelectOrganizationPath()`) to avoid 'unknown identifier' errors.
- **CSRF Tokens**: Automated API tests or HTML forms submitting POST/PUT/DELETE requests against Buffalo must include the token in the body as `authenticity_token`. For Svelte/JSON client requests, extract the token from the `<meta name="csrf-token">` tag and pass it via the `X-CSRF-Token` header.
- **Database Transaction Lifecycle**: Buffalo automatically wraps request handlers in a database transaction via the `popmw.Transaction` middleware. The transaction will be committed if the action returns `nil` (no error) and rolled back if it returns an `error`. Never call `tx.TX.Commit()` or `tx.TX.Rollback()` manually inside actions.
- **Environment Variables**: Always use `github.com/gobuffalo/envy` (e.g. `envy.Get("VAR", "default")`) instead of standard Go `os.Getenv` to read configuration so that local `.env` variables are correctly loaded and resolved.
- **Database CLI Fallback**: In environments where the `buffalo` CLI isn't functioning correctly for database tasks, you can use the `soda` CLI (e.g., `soda create -a`, `soda migrate`). If missing, it can be installed via `go install github.com/gobuffalo/pop/v6/soda@latest`.
- **Browser Testing & Seeds**: When running E2E/browser tests (such as with the `browser_subagent` tool) or verifying features locally, a default Super Administrator account is seeded by the migrations:
  - **Email / Username**: `admin@example.com`
  - **Password**: `password123`
  - **Organization**: `Default Organization`
  Use these credentials whenever authenticating during browser testing.

## UI/UX Standards (Non-Negotiable)

- **DaisyUI components first**: Use DaisyUI's built-in classes for all UI elements (buttons, cards, forms, modals, tables, alerts, etc.). Do not reinvent what DaisyUI already provides.
- **Minimalist CSS**: Write as little custom CSS as possible. Rely on DaisyUI and Tailwind utility classes. Only write custom CSS when a utility class genuinely does not exist for the need.
- **No inline styles**: Never use `style="..."` attributes. All styling must go through Tailwind/DaisyUI classes.
- **No generic colors**: Use DaisyUI semantic color tokens (`primary`, `secondary`, `accent`, `base-100`, `base-content`, etc.). Never hardcode hex values or use Tailwind's raw color palette (e.g. `bg-blue-500`).
- **Responsive by default**: Every page and component must work on mobile (≥320px) and desktop. Use DaisyUI responsive prefixes (`sm:`, `md:`, `lg:`) when layout needs to differ. Default to a single-column mobile layout and expand to multi-column on larger screens.
- **Dark mode**: The default DaisyUI theme is `dark` (set on `<html data-theme="dark">`). Do not hardcode light-mode assumptions.
- **Semantic HTML**: Use appropriate HTML5 elements (`<nav>`, `<main>`, `<section>`, `<article>`, `<header>`, `<footer>`) — do not `<div>` everything.
- **Accessibility**: Interactive elements must have accessible labels (`aria-label`, `<label for>`, descriptive button text). Avoid icon-only buttons without a title or aria-label.

## Backend Guidelines

- **Routes**: Define all routes in `actions/app.go`.
- **Auth**: Use Buffalo's middleware for session checks. Admin routes must use an `AdminRequired` middleware.
- **JSON API**: API endpoints (for Svelte) live under `/api/` and return JSON. Use `c.Render(200, r.JSON(...))`.
- **Migrations**: Always generate migrations with `buffalo pop generate fizz <name>` — never edit the DB schema manually.
- **Workers**: Use Buffalo `worker` for background processing (e.g., email delivery, heavy data processing).
- **Mailers**: Use Buffalo's built-in mailer generators and integrate with `worker.Simple` for asynchronous delivery.

---

## Frontend Guidelines

- **Svelte 5**: Use Runes syntax (`$state`, `$derived`, `$effect`). Do not use the legacy Options API.
- **API calls**: Use the native `fetch` API. Always handle loading (`isLoading`) and error (`errorMsg`) states explicitly in the UI.
- **Component size**: Keep Svelte components small and focused. Extract reusable pieces into `assets/js/components/`.
- **Mounting Svelte**: The application compiles Svelte assets into a single entry point [main.js](assets/js/main.js). To add a new Svelte component/page, define a target container in the Plush HTML template (e.g., `<div id="my-new-app"></div>`), import your component in `main.js`, and mount it targeting that element ID.

## Code Quality Standards

### Go

- **Formatting**: Always run `gofmt -w .` after any Go file change. No unformatted code is ever committed.
- **Error handling**: Every error return must be checked. Never use `_` to discard errors silently.
- **Naming**: Follow Go conventions — `CamelCase` for exported identifiers, `camelCase` for unexported. Acronyms stay uppercase (`userID`, `httpClient`).
- **Controllers stay thin**: Buffalo actions must only parse input, call a model/service function, and render a response. Business logic belongs in `models/`.
- **No global mutable state**: Avoid package-level variables that are mutated after `init()`, except for the render engine (`r`) and sync-guarded caches.
- **Imports**: Group imports as stdlib → external → internal, separated by blank lines. Use `goimports` if available.
- **Tests**: New model or service logic must have a corresponding `_test.go` file. Since test suites extend `ActionSuite` or `ModelSuite`, call assertions directly on the suite receiver (e.g. `as.NoError(err)`, `as.Equal(x, y)`) rather than importing the raw `testify/assert` package.

### JavaScript / Svelte

- **Svelte 5 Runes only**: Use `$state`, `$derived`, `$effect`, `$props`. Never use the Options API (`export let`, `onMount` stores, etc.).
- **No `var`**: Use `const` by default; `let` only when reassignment is required.
- **No unused variables or imports**: Remove dead code before committing.
- **Error and loading states**: Every `fetch` call must handle at minimum a loading boolean and an error message displayed to the user.
- **No inline event handlers in templates**: Extract handler logic into named functions in the `<script>` block.
- **Prop naming**: Use `camelCase` for props and event names.
- **One component per file**: Each `.svelte` file exports exactly one component.

### HTML / Plush Templates

- **Semantic elements**: Use `<nav>`, `<main>`, `<section>`, `<header>`, `<footer>`, `<article>` appropriately. Reserve `<div>` for purely structural/layout grouping with no semantic meaning.
- **DaisyUI classes over custom markup**: Use DaisyUI's component classes (e.g. `card`, `btn`, `form-control`, `input`, `table`) instead of building equivalent markup from scratch.
- **Responsive layout classes**: Every page layout must include responsive column classes (e.g. `grid grid-cols-1 md:grid-cols-2`, `flex flex-col md:flex-row`). No fixed-width pixel values.
- **No inline styles**: All styling via Tailwind/DaisyUI classes only.
- **Unique IDs**: Form inputs must have `id` attributes that match their `<label for>` values. IDs must be unique within the page.
- **CSRF**: All non-GET forms must include `<%= authenticity_token %>` via Buffalo's helper.
- **Flash messages**: Always render `<%= partial("flash.html") %>` at the top of the content area so user feedback is visible.

### CSS

- **Minimalist by default**: Do not write custom CSS unless a Tailwind utility or DaisyUI class genuinely cannot achieve the result. If you find yourself writing more than ~5 lines of custom CSS, reconsider whether a DaisyUI component or Tailwind utility covers the need.
- **No `!important`**: Its use is a signal that specificity is being fought — restructure the markup or classes instead.
- **No hardcoded colors**: All colors must reference DaisyUI's CSS custom properties (e.g. `hsl(var(--p))` for primary) or Tailwind semantic tokens.
- **Scoped to component**: Custom CSS written inside a Svelte `<style>` block is preferred over global rules; it is automatically scoped to the component.
- **Global CSS file** (`assets/css/application.css`): Reserved for `@import "tailwindcss"`, DaisyUI plugin registration, and truly global resets only. Do not add page-specific rules here.
